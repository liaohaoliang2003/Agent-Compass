# Agent Compass

**为项目建立一份 Agent 可以真正遵循的导航地图。**

Agent Compass 是一个 Codex Skill，用来在项目启动或重组时，帮你把“项目规则、工具分工、使用边界、输出位置、验证标准”整理成未来 Agent 能稳定读取和执行的工作契约。

它尤其适合这样的时刻：项目刚开始、工具很多、资料很散、团队规则还没成形，或者你希望后续 Agent 不再反复问同样的问题、不再混用 Skill、不再凭感觉改动项目。

## 它解决什么问题

很多项目不是缺少 Agent 能力，而是缺少一份清楚的“项目导航规则”。没有这层规则时，Agent 容易遇到这些问题：

- 不知道哪些文档、代码、数据或讨论才是权威来源。
- 多个 Skill/MCP/tool 职责重叠，导致工作流混乱。
- 不清楚什么能改、什么不能改、什么时候必须先问用户。
- 输出文件散落在项目根目录，后续很难追踪。
- 后来的 Agent 没读到前人总结，又从零开始猜项目背景。

Agent Compass 的目标，就是在项目早期把这些边界一次性立好。

## 适合哪些项目

Agent Compass 不只服务研究或论文项目。它适用于任何需要长期协作、工具编排或上下文沉淀的项目，例如：

- 软件仓库：Web app、CLI、后端服务、前端原型、库和工具包。
- 数据项目：数据分析、指标体系、实验记录、数据处理流水线。
- 研究项目：论文、毕业设计、文献综述、实验复现、学术图表。
- 内容项目：课程材料、知识库、报告、写作项目、团队文档。
- 产品项目：PRD、原型、用户研究、运营自动化、跨角色协作。

## 它会产出什么

使用本 Skill 后，Agent 通常会帮助你生成或更新这些内容：

- `AGENTS.md` 入口规则：让后续 Agent 知道进入项目后先读哪里。
- Agent master guideline：项目身份、核心事实、禁止事项、工作原则和验证标准。
- 项目级 Skill/MCP/tool policy：定义每个工具在本项目中负责什么、不负责什么、何时停下来问用户。
- 稳定输出目录：例如 `docs/agent_guidelines/`、`docs/agent_policies/`、`docs/project_context/`、`docs/project_checks/`。
- 针对特殊项目的附加边界：例如研究项目中的引用、CNKI、Zotero、私有数据、下载和版权规则。

## 快速使用

在 Codex 中可以这样调用：

```text
Use $agent-compass 为这个项目建立 Agent guideline、工具 policy 和 AGENTS.md 入口规则。
```

也可以直接描述你的目标：

```text
请帮我为这个软件仓库初始化 Agent 工作规则，明确哪些工具负责开发、测试、文档和发布，并写入 AGENTS.md。
```

```text
请为这个数据分析项目制定 Skill/MCP 分工，说明哪些数据源可信、输出保存到哪里、完成前需要哪些验证。
```

```text
请为这个论文项目建立 Agent 使用约束，明确文献工具、写作 Skill、引用证据和导师反馈的优先级。
```

## 推荐工作流

1. 让 Agent Compass 先读取项目现状：仓库结构、已有文档、配置、脚本、旧 `AGENTS.md`。
2. 回答少量高影响问题：项目目标、权威资料、可用工具、禁止事项、验收标准。
3. 让它盘点已有 Skill/MCP/tool，而不是一上来安装新工具。
4. 为重要工具写项目级 policy，避免多个工具互相抢主责。
5. 写入简洁的 Agent master guideline 和根目录 `AGENTS.md` 入口。
6. 后续任务让 Agent 先读这些规则，再执行具体开发、研究、写作或数据处理。

## 为什么叫 Agent Compass

因为它不是替你完成某一个具体任务的“锤子”，而是给整个项目定方向的“罗盘”。

它帮助 Agent 回答三个最关键的问题：

- 我现在在哪个项目里？
- 我应该相信哪些信息、使用哪些工具？
- 我完成工作前必须留下什么证据？

方向清楚之后，后续每个 Agent 才能更快、更稳、更少返工。

## 一句话总结

**Agent Compass 让每个项目在开始动手前，先拥有一份未来 Agent 看得懂、守得住、能复用的工作契约。**
